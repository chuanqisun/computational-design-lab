# 2026-03-12 Task Inventory For New Task-Centric App

This document compiles candidate user tasks from three sources:

- the design interview spec in `docs/20260312-design-interview.md`
- the canvas experience in `src/canvas-page.ts`
- the studio experience in `src/studio-page.ts`

The goal is to let a human editor prune, merge, rename, and prioritize tasks before implementation begins.

## Shared implementation model

Use a common `card` data structure across the app wherever possible.

- Core fields:
  - `title`
  - `description`
  - `imagePrompt`
  - `image`
  - `metadata`
- `metadata` should stay compact and purposeful. It can act as a trait system for task-specific behavior such as source provenance, selected scan lens, prompt revision history, structured attributes, render settings, or task state.
- Most tasks should accept cards as input and produce cards as output.
- Raw text, uploaded photos, generated concepts, simulated users, renders, and structured prompt artifacts should all be representable as cards or card collections.
- The main implementation goal is not to build many unrelated task types, but to build a few reusable task patterns over cards.

## Three high-level task types

The app should treat most user flows as one of these three types:

1. `lift` — convert unstructured input into cards
   - Examples: import a photo, write a note, scan a product photo into attributes, convert a reference into concept cards
2. `transform` — generate more cards from cards
   - Examples: blend references, generate concepts, revise prompts, simulate users, stage scenes, render outputs
3. `view` — arrange, filter, and select cards
   - Examples: organize a board, browse prior work, switch tasks, inspect generated assets, curate a selection for the next step

This framing should drive implementation toward shared card input/output handling, shared task chrome, and reusable persistence rather than bespoke flows for every feature.

## How to edit this list

- Keep the tasks that should become first-class app experiences.
- Merge tasks that are really sub-steps of a larger flow.
- Mark anything as `defer` if it should wait until after the prototype.
- Preserve prompt wording from the existing apps wherever possible during implementation.

Suggested status labels:

- `keep` — definitely implement as a task
- `merge` — combine into another task
- `defer` — not for first implementation round
- `drop` — do not carry forward

## 1. High-level product requirements extracted from the interview

These are not tasks by themselves, but they constrain how tasks should work.

1. The app is for industrial designers who are not comfortable with prompting but are strong in product design.
2. The app should be a prompt-building and prompt-running environment.
3. The UI should be task-centric: one focused task fills the screen.
4. Task creation and task switching should happen in a modal flow.
5. Multiple tasks can exist in parallel and remain resumable.
6. Inputs, generated assets, and credentials should persist locally.
7. Prompt state should eventually be shareable by URL when feasible.
8. Errors should be visible, non-destructive, and retryable.
9. Users should be able to run multiple generations without the system blocking them.
10. Use intrinsic HTML behavior, TypeScript, RxJS, lit-html, and Vite.
11. Prefer minimal visual styling, little color, no emoji, and efficient use of large-screen space.
12. The system should offer a generic “improve my prompt” or prompt-revision capability.
13. Nearly all task state should be representable as cards plus lightweight task-specific metadata.
14. Tasks should compose by passing selected cards into the next task whenever possible.

## 3. Shared task model implications

Before detailing each task, the implementation should assume these common behaviors:

- Every task operates on a current card collection plus an optional current selection.
- Every task can declare:
  - accepted card traits in `metadata`
  - produced card traits in `metadata`
  - whether it consumes raw input, cards, or both
- `lift` tasks create cards from raw input.
- `transform` tasks read cards and append new derived cards rather than mutating prior outputs destructively.
- `view` tasks primarily manipulate presentation, filtering, grouping, and task navigation state over cards.
- Cross-task transitions should pass card references and selection state, not bespoke payload formats.
- Prompt-like artifacts should still be cards, even when their most important payload lives in `description` or `metadata`.

## 4. Detailed task breakdown

### T01 — Capture reference material

- Source evidence:
  - Canvas has a dedicated capture tool with upload and camera capture.
  - Studio has a scan flow for product photos.
- User value:
  - Quickly import reference imagery without leaving the app.
- Inputs:
  - Camera capture
  - File upload
  - Clipboard paste
  - Imported references from another app state
- Outputs:
  - Reference cards with image fields and source metadata
- Notes:
  - This is a foundational `lift` task and should create cards directly.

### T02 — Write textual inspiration

- Source evidence:
  - Canvas writer flow adds text cards.
- User value:
  - Add designer intent, labels, or raw prompt fragments.
- Inputs:
  - Freeform text
  - Optional distinction between title and body
- Outputs:
  - Text cards usable by downstream tasks

### T03 — Organize and curate cards

- Source evidence:
  - Canvas supports selecting, moving, z-ordering, focusing, gathering, deleting, and deselecting items.
- User value:
  - Curate card collections before asking AI to interpret or transform them.
- Inputs:
  - Cards
  - Selection gestures
- Outputs:
  - Refined arrangement, filters, groups, and selection state
- Notes:
  - This is the main `view` pattern and may remain a dedicated workspace rather than a modal task.

### T04 — Scan references into concept cards

- Source evidence:
  - Canvas conceptual scan offers scan modes such as emotions, associations, symbolism, shape, material, color, tactility, acoustics, interactions, and narrative.
- User value:
  - Turn selected reference cards into interpretable design language.
- Inputs:
  - Selected cards
  - Chosen scan lens
- Outputs:
  - New concept cards with titles, descriptions, and scan metadata
- Open questions:
  - Should each scan lens remain separate traits, or become one task with switchable sub-modes?

### T05 — Visualize cards

- Source evidence:
  - Canvas visualize flow exposes at least abstract and concrete modes.
- User value:
  - Translate text ideas into visual directions.
- Inputs:
  - Selected text-bearing cards
  - Visualization mode
- Outputs:
  - New image-prompt cards
- Recommendation:
  - Likely merge into a broader `transform cards` family with task presets.

### T06 — Blend cards

- Source evidence:
  - Canvas blend flow requires multiple selected items and accepts blending instructions.
- User value:
  - Explore hybrids and recombinations across references.
- Inputs:
  - Multiple selected cards
  - Optional blend instruction
- Outputs:
  - New blended cards

### T07 — Generate design concept cards

- Source evidence:
  - Canvas design flow asks for design requirements and count.
- User value:
  - Produce multiple design directions from existing inspiration and constraints.
- Inputs:
  - Selected cards
  - Requirements text
  - Desired number of concepts
- Outputs:
  - New concept cards with titles, bodies, and image prompts
- Notes:
  - This appears close to the interview’s “brainstorming ideas” task.

### T08 — Iterate on cards

- Source evidence:
  - Canvas iterate flow accepts optional feedback and a count.
- User value:
  - Generate variants without rebuilding context from scratch.
- Inputs:
  - Selected cards
  - Optional direction or feedback
  - Desired number of outputs
- Outputs:
  - Revised variant cards

### T09 — Simulate user feedback

- Source evidence:
  - Canvas user testing flow asks for trait, segment, and user count.
- User value:
  - Get qualitative reactions from simulated target users.
- Inputs:
  - Selected concept cards
  - Trait focus
  - User segment
  - Number of simulated users
- Outputs:
  - Persona cards and feedback cards
- Notes:
  - This aligns with the interview’s “simulated users” example task.

### T10 — Render cards into imagery

- Source evidence:
  - Canvas render flow accepts a prompt and optionally appends selected card text.
- User value:
  - Turn prompt-bearing cards into image outputs.
- Inputs:
  - Selected cards
  - Optional render instruction
- Outputs:
  - Generated image cards

### T11 — Hand off selected cards to another task

- Source evidence:
  - Canvas export opens selected items in Studio.
  - Studio imports via `fromCanvasBlob` and seeds scanned photos plus custom instructions.
- User value:
  - Move from one task to another without translating data into a different format.
- Inputs:
  - Selected cards
- Outputs:
  - Imported task session seeded with the same cards and additional task metadata
- Notes:
  - This should be treated as shared platform behavior rather than a highly specialized task.

### T12 — Create attribute cards from a library

- Source evidence:
  - Studio left panel exposes shapes, materials, surface options, mechanisms, and colors.
- User value:
  - Build prompts through domain-specific structured choices instead of freeform prompting.
- Inputs:
  - Library selections
  - Filter text
- Outputs:
  - Attribute cards or card traits representing the chosen structure
- Notes:
  - This is likely one of the core MVP tasks for packaging designers.

### T13 — Scan product photos into attribute cards

- Source evidence:
  - Studio scan uses AI to infer shapes, materials, mechanisms, and colors from uploaded product photos.
- User value:
  - Start from existing design references instead of manually selecting every attribute.
- Inputs:
  - One or more product photos
- Outputs:
  - Auto-populated attribute cards and source photo cards
- Notes:
  - This can be positioned as “analyze an existing package” or “extract prompt ingredients from reference image.”

### T14 — Synthesize prompt card

- Source evidence:
  - Studio synthesis combines selected attributes, scanned photos, and optional custom instructions into XML.
- User value:
  - Assemble a production-ready prompt card from modular design decisions.
- Inputs:
  - Selected attribute cards
  - Optional custom instructions
  - Optional scanned photo cards
- Outputs:
  - Prompt card with structured prompt content in `description` or `metadata`
- Notes:
  - This appears to be the clearest expression of the interview’s core “guided prompt builder” job.

### T15 — Revise prompt card

- Source evidence:
  - Studio revise uses prior conversation history and edit instructions to update XML.
- User value:
  - Improve the current prompt while preserving previous work.
- Inputs:
  - Existing prompt card and synthesis history
  - Edit instructions
- Outputs:
  - Updated prompt card
- Notes:
  - This directly satisfies the interview requirement for a generic prompt-improvement action.

### T16 — Stage scene cards

- Source evidence:
  - Studio photo booth accepts scene text and suggested scenes derived from selected mechanisms.
- User value:
  - Re-contextualize a product concept into usage or marketing scenes.
- Inputs:
  - Existing prompt card
  - Photo scene text or suggested scene
- Outputs:
  - Scene-specific prompt cards stored for rendering

### T17 — Generate product image cards

- Source evidence:
  - Studio gallery renders images from generated photo prompts.
- User value:
  - See packaging concepts as rendered imagery.
- Inputs:
  - Scene-specific prompt cards
- Outputs:
  - Generated image cards in a persistent collection
- Notes:
  - This may be merged with T16 as one “photo render” task if a simpler MVP is needed.

### T18 — Generate product animation cards

- Source evidence:
  - Studio gallery supports animation instructions and sound description for video generation.
- User value:
  - Explore motion or product interaction in a more expressive medium.
- Inputs:
  - Existing generated image cards
  - Animation instructions
  - Sound description
- Outputs:
  - Generated video cards
- Recommendation:
  - Likely defer for prototype unless motion is a primary stakeholder need.

### T19 — Edit generated output cards

- Source evidence:
  - Studio gallery lets the user edit XML for a generated output and re-run the result.
- User value:
  - Make targeted changes after a render exists.
- Inputs:
  - Existing output cards
  - Manual edits
- Outputs:
  - New edited output cards
- Recommendation:
  - Strong feature, but can be deferred if prompt revision at the synthesis stage already covers most needs.

### T20 — Manage API connections

- Source evidence:
  - Both apps register Gemini-powered features and expose setup dialogs.
- User value:
  - Enable live AI execution with the user’s own API key.
- Inputs:
  - Gemini API key
- Outputs:
  - Persisted connection state
- Notes:
  - This is infrastructure, but must be visible in the task system.

### T21 — Resume prior work

- Source evidence:
  - Both apps persist substantial state locally.
  - The interview explicitly requires resumable flows.
- User value:
  - Return after refresh without losing task context.
- Inputs:
  - Persisted task state and card collections
- Outputs:
  - Restored task session

### T22 — Switch between tasks

- Source evidence:
  - The interview explicitly calls for modal task creation and task switching, with multiple simultaneous tasks.
- User value:
  - Jump between workstreams while keeping each one intact.
- Inputs:
  - Task list or task switcher selection
- Outputs:
  - Active task view changes with its card collection restored
- Notes:
  - This is a shell requirement, but it is central enough to track as a task-platform capability.
